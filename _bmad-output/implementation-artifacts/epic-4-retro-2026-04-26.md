# Epic 4 Retrospective (Customer Web App)

## Epic Review

### Scope reviewed

- `4-1-app-shell-navigation`
- `4-2-template-browse-run-ui`
- `4-3-accessibility-baseline`

### What went well

- Re-entry from the UI freeze was smooth: the imported UI kit became a practical baseline rather than throwaway work.
- App shell navigation was normalized into clear top-level routes (`templates`, `runs`, `assistant`, `builder`) with no dead-end destinations.
- Template browse/run flows were connected to real workflow API endpoints while preserving deterministic fallback behavior for demo resilience.
- Accessibility baseline improvements were delivered quickly across focus visibility, keyboard access, and semantic landmarks.
- Lightweight smoke checks helped catch and prevent route-shell regressions during rapid UI iteration.

### What was challenging

- Balancing API-first integration with fallback behavior introduced extra state paths and edge cases.
- Route transitions and nested views (Templates -> Runner) needed additional hardening to avoid confusing navigation resets.
- Accessibility changes had to be layered over existing inline-style UI patterns, which increases implementation verbosity.

### Key lessons

- A frozen baseline can accelerate delivery if reopened with a story-by-story integration plan.
- API-backed UI should keep deterministic fallback paths in MVP phases to avoid blocked demos and testing.
- Accessibility work is most efficient when included continuously rather than deferred to a final pass.
- Small smoke tests around shell wiring provide strong value for low effort in UI-heavy iterations.
