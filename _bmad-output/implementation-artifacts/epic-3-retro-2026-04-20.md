## What Went Well

- Reliability work was split into clear vertical slices (status model, webhook idempotency, mock determinism).

- Test coverage expanded with focused integration tests and regression checks tied directly to acceptance criteria.
- Code review findings were resolved promptly and documented in story records.

## Challenges / Friction

- Determinism work required balancing stable outputs with uniqueness across sequential mock calls.
- Retrospective tracking occasionally lagged implementation status updates, requiring explicit closeout sync.
- Error-prone broad exception catches were present initially in webhook processing and had to be narrowed.

## Key Learnings

1. Canonical status behavior should be centralized early to prevent persistence inconsistencies.
2. Idempotency guarantees are strongest when backed by DB constraints and explicit duplicate-path tests.
3. CI stability depends on deterministic mocks, including IDs and timestamps, not just mock mode flags.
