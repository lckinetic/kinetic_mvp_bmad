# Epic 5 Retrospective (E5 Assistant)

## Epic Review

### Scope reviewed

- `5-1-chat-session-proposal`
- `5-2-confirm-before-execute`
- `5-3-ui-handoff`

### What went well

- Assistant backend was delivered incrementally with clear API contracts per story.
- Story slicing was effective: proposal creation -> confirmation gate -> UI handoff contract.
- Structured error envelope consistency held across new assistant routes.
- Tests were expanded each step and stayed deterministic in mock/dev paths.
- Follow-up hardening happened early (DB-backed persistence, idempotent behavior, branch-aware handoff mapping) instead of being deferred.

### What was challenging

- Initial implementation started with in-memory proposal storage and required a follow-up migration to DB-backed persistence.
- State transitions (`proposed`, `confirmed`, `executed`) required repeated refinement to preserve idempotency and guardrails.
- Handoff payload mapping needed additional iteration to preserve explicit branch topology rather than only linear links.
- Local test execution required strict use of the project `.venv` due to global interpreter dependency gaps.

### Key technical lessons

- For assistant workflows, persistence should default to DB-backed models as soon as cross-endpoint state is introduced.
- Confirmation-first gating is best implemented at service boundary and surfaced via explicit machine-readable API errors.
- Handoff contracts should encode graph semantics explicitly (`next`, `next_steps`) and only fallback to linear mapping when absent.
- OpenAPI checks plus API behavior tests together prevented contract drift.
